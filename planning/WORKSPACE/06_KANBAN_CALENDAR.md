## Overview

Dual-feature backend module that drives the dealer workspace's visual pipeline board (Kanban) and event scheduling (Calendar). The Kanban board is the operational heart of the vehicle lifecycle: every drag-and-drop is a validated state-machine transition. The Calendar captures structured events — both human-scheduled and auto-generated by lifecycle triggers.

## State Machine

11 pipeline stages (columns) map 1:1 to vehicle states:

```
sourcing → acquired → reconditioning → photography → listed
                   ↘                              ↙
                    listed → inquiry → negotiation → reserved → sold → in_transit → delivered
```

`ValidateTransition(from, to string) error` enforces the transition table. Every `MoveCard` call validates the transition, rejects illegal moves with 422 Unprocessable Entity.

---

## WIP Limits

Each column carries a `vehicle_limit` (0 = unlimited). `MoveCard` counts existing cards in the target column before inserting. If `count >= vehicle_limit`, returns 422 with message `"column {name} is at WIP limit ({n})"`.

---

## Auto-Generated Calendar Events

`OnVehicleStateChange(ctx, tenantID, vehicleID, newState)` fires on every successful `MoveCard`:

| Trigger State | Event Type | Offset |
|---|---|---|
| `in_transit` | `transport_delivery` | +3 days |
| `reserved` | `registration` | +5 days |

Events are marked `auto_generated=true`. The function is also exposed directly on the Store for external CRM callers.

## API Endpoints

| Method | Path | Description |
|---|---|---|
| GET | `/api/v1/kanban/columns` | List all columns with nested cards |
| POST | `/api/v1/kanban/columns` | Create custom column |
| PUT | `/api/v1/kanban/columns/{id}` | Patch column (name/color/limit) |
| PUT | `/api/v1/kanban/cards/{vehicleId}/move` | Move card (state transition) |
| PUT | `/api/v1/kanban/cards/{vehicleId}` | Patch card metadata |
| GET | `/api/v1/calendar/events?from=&to=` | List events in date range |
| POST | `/api/v1/calendar/events` | Create event |
| GET | `/api/v1/calendar/events/upcoming?days=` | Next N days (default 7) |
| PUT | `/api/v1/calendar/events/{id}` | Edit event |
| DELETE | `/api/v1/calendar/events/{id}` | Cancel event |

All endpoints require `X-Tenant-ID` header.

## Design Decisions

**SQLite for storage:** consistent with all other workspace modules; pure-Go `modernc.org/sqlite`, no CGo.

**Tenant isolation via header:** `X-Tenant-ID` is the sole tenancy signal; all queries are scoped to it. `InitTenant` is idempotent — creates the 11 default columns if absent.

**State key on columns:** default columns carry a `state_key` matching the vehicle state machine. Custom columns added by dealers have no `state_key` and bypass state-machine validation — they're treated as free staging areas.

**Overdue metric:** `metricCalendarOverdue` is a snapshot gauge updated at query time by `UpcomingEvents` and `ListEvents`. Intended for Prometheus alerting on stale scheduled events.
