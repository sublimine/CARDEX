# E12 — Edge Dealer Push

**Sprint:** 28  
**Status:** MVP complete (gRPC server + Tauri scaffold)  
**Priority:** 1500 (highest — dealer-originated data)  
**Branch:** `sprint/28-edge-tauri-grpc`

---

## Overview

E12 is the highest-priority extraction strategy.  Instead of scraping a dealer's
website, the dealer installs a lightweight Tauri desktop client on the same
machine as their DMS (CDK Global, Reynolds & Reynolds, etc.).  The client reads
the DMS database and pushes vehicle records directly to CARDEX via gRPC.

**Why this is priority 1500:**
- Data originates from the dealer's own DMS — no scraping heuristics or field
  mapping ambiguity.
- VIN is a first-class field (not inferred from URLs or images).
- Pricing is in cents (exact integer) with explicit currency.
- The dealer controls what they publish — implicit consent for processing.

```
│  Tauri App (Rust)       │  ──── gRPC TLS ──→  edge-push-server :50051

                                          Extraction orchestrator
                                           (E12 reads staging)
                                                   ↓
                                           Quality pipeline (V01-V20)
                                                   ↓
                                           KG vehicle_record table
```

## Server-side components

### gRPC service (`extraction/api/edgepb`)

| Method | Type | Description |
|--------|------|-------------|
| `PushListings` | client-streaming | Receives batches of `ListingBatch`, returns `PushResponse` |
| `Heartbeat` | unary | Returns server Unix timestamp |

Authentication: `dealer_id` + `api_key` embedded in each `ListingBatch`.
The api_key is validated against the SHA-256 hash in `edge_dealers` table.

### Database tables

```sql
-- edge_dealers: registered dealer accounts
CREATE TABLE edge_dealers (
  dealer_id    TEXT PRIMARY KEY,  -- ULID
  name         TEXT NOT NULL,
  country      TEXT NOT NULL,     -- ISO-3166-1 alpha-2
  vat_number   TEXT NOT NULL,
  api_key_hash TEXT NOT NULL,     -- SHA-256 hex
  created_at   TEXT NOT NULL,     -- RFC3339
  revoked_at   TEXT,              -- NULL = active
  vies_verified INTEGER NOT NULL DEFAULT 0
);

-- edge_inventory_staging: inbound push queue
CREATE TABLE edge_inventory_staging (
  id            TEXT PRIMARY KEY,
  dealer_id     TEXT NOT NULL REFERENCES edge_dealers(dealer_id),
  vehicles_json TEXT NOT NULL,
  received_at   TEXT NOT NULL,
  consumed      INTEGER NOT NULL DEFAULT 0,
  consumed_at   TEXT
);
```

### Rate limiting

1000 listings per dealer per 60-second sliding window.
Over-limit listings are rejected with `reason: "rate_limit_exceeded"` in `PushResponse`.

### Metrics

| Metric | Type | Labels |
|--------|------|--------|
| `cardex_edge_push_listings_total` | Counter | `dealer`, `status={accepted,rejected}` |
| `cardex_edge_push_latency_seconds` | Histogram | `dealer` |

## Dealer onboarding CLI

**Binary:** `cardex-dealer` (`extraction/cmd/cardex-dealer/`)  
**Environment:** `EDGE_DB_PATH` (default: `./data/discovery.db`)

```bash
# Register (with VIES validation for EU dealers)
cardex-dealer register --name "AutoHaus Berlin" --country DE --vat DE123456789

# Skip VIES (for testing or non-EU countries)
cardex-dealer register --name "Garage Test" --country CH --vat CHE123456789 --skip-vies

# List all dealers
cardex-dealer list

# Revoke (immediate — existing api_key rejected on next push)
cardex-dealer revoke 01JFXYZABCDE1234567
```

The plaintext api_key is shown **once** at registration.  It is never stored
in the DB (only the SHA-256 hash).

### VIES integration

For DE/FR/ES/BE/NL dealers, the CLI calls the VIES REST API before registration:

```
GET https://ec.europa.eu/taxation_customs/vies/rest-api/ms/{CC}/vat/{NUMBER}
```

If VIES returns `isValid: false`, the dealer is still registered but
`vies_verified = 0`.  The operator must decide whether to proceed.  CH dealers
use the UID-Register (SOAP) — VIES is skipped.

## Data flow detail

1. Tauri client sends `PushListings` stream (one or more `ListingBatch` messages).
2. Server authenticates dealer_id + api_key on first batch.
3. Server applies rate limit (1000/min sliding window).
4. Server validates each listing: VIN 17 chars, Make/Model/Year present.
5. Accepted listings written to `edge_inventory_staging` (JSON).
6. Listings also persisted directly via `storage.PersistVehicles()` (fast path).
7. Extraction orchestrator's E12 strategy also reads unconsumed staging rows on
   each orchestration cycle (slow path / eventual consistency fallback).
8. Vehicles tagged with `AdditionalFields["extraction_method"] = "e12_edge_push"`.
9. `PushResponse` returned with accepted/rejected counts and per-VIN rejection reasons.

## Proto generation (`make proto`)

See `Makefile` target `proto`.  Requires `protoc` and `protoc-gen-go-grpc`.

Until `make proto` is run, the Go server uses hand-written types in
`extraction/api/edgepb/` that are wire-compatible with the proto definition.
The swap to generated types is a drop-in replacement (same struct layout).

## Security notes

- TLS 1.3 minimum on the gRPC server (`EDGE_INSECURE=true` for dev only).
- API keys are 32 random bytes (256 bits), stored as SHA-256 hex.
- Keys shown once at `cardex-dealer register`, never recoverable.
- Revocation is immediate — next push attempt returns `UNAUTHENTICATED`.
- Rate limit prevents inventory flooding attacks (1000/min/dealer).
- VIN validation (17 chars) prevents trivial injection attempts.
- PII sanitizer runs on accepted vehicles before `PersistVehicles()`.
